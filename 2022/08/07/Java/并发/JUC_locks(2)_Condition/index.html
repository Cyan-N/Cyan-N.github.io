J.U.C.locks(2)--AQS

本文最后更新于：2022年8月8日 晚上

简介

在locks包下有一个类是AbstractQueuedSynchronizer，其简写就是AQS，AQS是一个用来构建锁和同步器的框架，使用AQS能简单且高效地构造出应用广泛的大量的同步器。

AQS核心思想是，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态；如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制，这个机制AQS是用CLH队列锁实现的，即将暂时获取不到锁的线程加入到队列中。

CLH(Craig,Landin,and Hagersten)队列是一个虚拟的双向队列(虚拟的双向队列即不存在队列实例，仅存在结点之间的关联关系)。AQS是将每条请求共享资源的线程封装成一个CLH锁队列的一个结点(Node)来实现锁的分配。Node其实就是AQS里面的一个内部类

在AQS中，资源存在两种：独占资源和共享资源。最简单的例子就是，如果一个资源只能被读取而不能被修改，那么这个资源就可以作为共享资源。之前也提到AQS相当于是一个框架，自定义同步器在实现时只需要实现共享资源的获取与释放方式即可（其实也就是锁的最基本特性），至于具体线程等待队列的维护(如获取资源失败入队/唤醒出队等)，AQS已经在上层已经帮我们实现好了。

在我们自定义同步器时，继承AbstractQueuedSynchronizer并重写指定方法（就是对共享资源的获取和释放）。然后AbstractQueuedSynchronizer中的模板方法就会调用使用者重写的方法。与通过实现接口的方式区别很大，接口是通过多态降低类与类之间的耦合度，它只是定义许多行为，但不会定义一个模板。

Node类

之前我们提到AQS有一个CLH队列，其实就是Node节点形成了双向链表，该队列又叫同步队列–Sync queue。Node是AQS的一个内部类。

每个被阻塞的线程都会被封装成一个Node结点，放入队列。每个节点包含了一个Thread类型的引用，并且每个节点都存在一个状态

Condition接口

在Java API里面对Condition接口的解释如下

Condition factors out the Object monitor methods (wait, notify and notifyAll) into distinct objects to give the effect of having multiple wait-sets per object, by combining them with the use of arbitrary Lock implementations. Where a Lock replaces the use of synchronized methods and statements, a Condition replaces the use of the Object monitor methods.

大致意思是，之前我们使用synchronized关键字实现同步方法和语句，如果一个线程需要等待或者唤醒其他线程，就需要用到锁对象中的wait/notify方法。现在Condition代替了锁对象的方法，Lock代替了synchronized的使用。

所以在Condition中定义的方法也很简单，通过查看相关API一目了然。

ConditionObject

类核心方法

acquire方法

该方法以独占模式获取(资源)，忽略中断，即线程在aquire过程中，中断此线程是无效的。源码：

  • 首先调用tryAcquire方法，调用此方法的线程会试图在独占模式下获取对象状态。此方法应该查询是否允许它在独占模式下获取对象状态，如果允许，则获取它。在AbstractQueuedSynchronizer源码中默认会抛出一个异常，即需要子类去重写此方法完成自己的逻辑。
  • 若tryAcquire失败，则调用addWaiter方法，addWaiter方法完成的功能是将调用此方法的线程封装成为一个结点并放入Sync queue。
  • 调用acquireQueued方法，此方法完成的功能是Sync queue中的结点不断尝试获取资源，若成功，则返回true，否则，返回false。

分析addWaiter方法，需要判断同步队列是否初始化。addWaiter方法使用快速添加的方式往sync queue尾部添加结点，如果sync queue队列还没有初始化，则会使用enq插入队列中。

总结一下就是，addWaiter会将节点加入到队列尾部，如果队列没有初始化，那么初始化和节点加入的工作交给enq。两个函数都会返回尾节点。
