内部结构

该类有五个内部类

内部类Sync

Sync抽象类继承AQS抽象类

Sync类内部也存在两个内部类：HoldCounter和ThreadLocalHoldCounter。

HoldCounter

count表示某个读线程重入的次数，tid表示该线程的tid字段的值，该字段可以唯一标识一个线程。

ThreadLocalHoldCounter

ThreadLocalHoldCounter重写了ThreadLocal的initialValue方法，**ThreadLocal类可以将线程与对象相关联。**在没有进行set的情况下，get到的均是initialValue方法里面生成的那个HolderCounter对象。

理解计数器

这两个类很重要，都是与读锁相关的，强调：多个不同的线程可以同时获得一个读锁。其中，当一个线程获取了读锁，它就会分配一个HoldCounter用来记录它的获取锁的次数。多个线程同时获取读锁，就会产生多个HoldCounter，它们存放在ThreadLocalHoldCounter中，该类继承了ThreadLocal，Thread有一个属性就是ThreadLocal.ThreadLocalMap threadLocals = null;。关于threadLocals，之后会再开一个坑。现在你就只需要知道ThreadLocalHoldCounter用于存放HoldCouter即可。

计数器的初始化和改变发生在获取锁和释放锁的过程，所以此处我们先看一下获取读锁的源码（Sync源码），因为读锁是共享锁，所以是shared。

Sync的基本函数

  • 构造函数：在Sync的构造函数中设置了本地线程计数器和AQS的状态state。

对ReentrantReadWriteLock对象的操作绝大多数都转发至Sync对象进行处理，所以要对Sync类中的重点函数进行分析。

  • sharedCount函数：标识占有读锁的线程数量

SHARED_SHIFT是Sync重要的变量，值为16。直接将state右移16位，就可以得到读锁的线程数量，因为state的高16位表示读锁，对应的低十六位表示写锁数量。

  • exclusiveCount函数：表示占有写锁的线程数量

写锁占有的最大线程数量是(1 << 16) - 1， 直接将状态state和(2^16 - 1)做与运算，其等效于将state模上2^16。

写线程获取锁修改state，就是正常的+1，可以理解为低16位+1；而读线程获取锁修改state，是高16位+1，即为每次state+=SHARED_UNIT，SHARED_UNIT是一个很大数（1<<16+1），每次读锁state加这么大个数，怎么是+1呢，这里就要理解高16位+1是在二进制下+1。

重写AQS函数

为了减轻读者阅读压力，之后的方法不会给出所有源码，而是以流程图的形式，建议自己结合源码阅读。之前我们讲了exclusiveCount函数，配合getState获得写线程的数量

现在我们就可以学习重写AQS里的重要函数：

  • tryRelease函数：先检测当前线程是否是独占线程，再是否资源，之后会判断是否释放成功（完全释放，如果还有重入则不算成功），设置释放后的状态。

  • tryAcquire函数：从上面的内容可以看出，单独看c是没有多大意义的。所以要判断w的值。并发环境中，一个资源，如果所有线程都是读操作，那么就没必要加锁，如果存在写操作，那么读操作和写操作都要加锁。这也就是说，如果c != 0 && w == 0，说明只有读锁，而没有写锁，但是此时线程想要获取写锁也是不允许的。tryAcquire的大致流程就是这样的，先判断有没有读锁，在判断当前写锁的独占线程是不是自己，最后判断获取的锁线程数量会不会超过MAX_COUNT。

tryAcquire和tryRelease都是与写操作相关的，我们知道，AQS除了独占模式，还有共享模式，需要重写tryReleaseShared和tryAcquireShared函数。而若干个线程的读操作可以再共享模式中执行。

  • tryAcquireShared函数：如果该资源被别的线程的写锁获取，那么该线程就不能再获取读锁。也就是说，一个线程想要获取读锁，必须是两种前提之一：1是整个同步器都没有写线程；2就是哪怕有写线程，这个线程也必须是自己这个线程。该函数我们在讲计数器时也看了源码和注释，所以此处直接给出流程图。

此处不必过多追究threadLocals，之后会讲到的。

  • tryReleaseShared函数：一个线程释放读锁时，需要的改动其实就是HoldCounter，所以大致流程就是判断firstReader。接着修改自身的HoldCounter等，最后死循环直到成功CAS修改读锁状态。

锁的公平性

与ReentrantLock一样，该类的读写锁都是默认非公平的。

NonfairSync和FairSync继承Sync，实现的是两种阻塞策略——writerShouldBlock和readerShouldBlock。

NonfairSync

  • writerShouldBlock：写锁在抢锁前永远不会阻塞，非公平性
  • readerShouldBlock：读线程抢锁之前，如果队列head后继（head.next）是独占节点时阻塞。

FairSync

无论是写线程还是读线程，只要同步队列中有其他节点在等待锁，就阻塞，这就是公平性。

此时你再去阅读Sync重写的AQS方法，你就会对线程是否应该阻塞有更清晰认识。非公平锁在tryAcquire中始终不会管同步队列是否存在等待更久的节点，而是直接去CAS替换抢锁。
